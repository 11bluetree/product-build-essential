# Testing Strategy

## 方針

- 速いテストを中心に組み立てる
- ドメインルールは単体テストで守る
- 境界や外部接続は統合テストで守る
- 重要な仕様変更はテストを先に更新する

## テストの分担

### 単体テスト

- エンティティ
- 値オブジェクト
- 小さなユースケース
- 純粋関数

### 統合テスト

- API エンドポイント
- リポジトリ
- 認証、認可
- 外部サービス連携の主要経路

## テストケース命名

- 閾値や文言の細部に依存しすぎない
- 仕様の意図を名前にする
- 数字そのものではなく、最小未満、最大超過、必須欠落のように概念で表現する

## テストデータ

- 境界値テストは固定値を使う
- 通常系や網羅補助には Faker などの生成器を使う
- データ生成の責務はヘルパーに寄せる

## テスト環境の原則

- 本番に近い実行経路を優先する
- テスト間の独立性を守る
- セットアップとクリーンアップを標準化する
