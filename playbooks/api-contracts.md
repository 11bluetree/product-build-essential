# API Contracts

## 基本方針

- API は内部実装ではなく契約として扱う
- 入力と出力は型とスキーマで定義する
- 破壊的変更は意図的に管理する

## 契約設計のルール

- リクエスト型、レスポンス型、バリデーションスキーマを揃える
- DTO はシリアライズ可能な値のみで構成する
- ドメインモデルをそのまま外へ出さない
- 文字列、数値、日付、状態の表現方法を統一する

## 境界で検証するもの

- パスパラメータ
- クエリパラメータ
- リクエストボディ
- ヘッダ
- ファイル入力

## 変更管理

- 契約変更時はまずテストと型を更新する
- 互換性を壊す場合は移行方針を明示する
- クライアントとサーバの片側だけ先に変えても破綻しない段階的変更を優先する
