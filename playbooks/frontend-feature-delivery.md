# Frontend Feature Delivery

## 実装順序

フロントエンド機能は、次の順序で進めると責務が崩れにくい。

1. API 契約を確認する
2. フォームや画面固有のスキーマを定義する
3. API 通信関数を作る
4. 状態管理やイベント処理をフックやストアへ集約する
5. 再利用 UI を整える
6. 画面固有コンポーネントを組み立てる
7. ルートまたはページに統合する

## 原則

- 画面固有状態は UI 層で閉じる
- API 契約を直接 UI 状態と同一視しない
- 見た目の部品と状態管理を分離する
- データ取得と送信の責務を共通化する

## UX の最低基準

- 入力エラーは利用者に理解できる言葉で返す
- 非同期処理にはローディングと失敗時の導線を用意する
- 動的フォームや複雑な入力は状態の責務を 1 か所に寄せる

## レビュー観点

- ページコンポーネントが肥大化していないか
- 画面ロジックが再利用 UI に混入していないか
- API エラーの表示が実装者都合のまま露出していないか
