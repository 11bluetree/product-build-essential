# Product Development Principles

## 目的

プロダクト開発で繰り返し起きる失敗を避けるために、設計と実装の判断軸を先に定義する。

## 原則

### 1. 仕様より先に責務を整理する

- 画面、API、データベースから考え始めない
- まず「誰が」「何をしたいか」と「その結果どの状態が変わるか」を定義する
- 振る舞いの中心にある概念を先に言語化する

### 2. ドメイン知識を技術選定から守る

- ビジネスルールはフレームワークやインフラの都合に引きずられない場所へ置く
- 保存方法、通信方法、認証方式が変わっても成立する設計を優先する

### 3. 境界を明確にする

- 外部入力は必ず境界で検証する
- 画面用の状態、API契約、永続化モデル、ドメインモデルを同一視しない
- 変換が必要なら明示的に変換する

### 4. 実装順序を標準化する

- 新機能は同じ順序で進める
- 機能ごとに設計の型が揃うとレビュー、保守、オンボーディングが容易になる

### 5. 技術選定は理由まで残す

- 採用技術だけでなく、何を捨てたかも記録する
- 将来の再評価条件も残す

## 避けるべき状態

- ドメインルールがコントローラや UI に散在する
- API の入出力と内部モデルが無秩序に混ざる
- 例外処理、認証、検証が実装者依存になる
- 技術判断が口頭や暗黙知のまま消える
