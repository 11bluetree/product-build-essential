# Architecture Principles

## 推奨する基本構造

以下のような責務分離を基準にする。

1. Presentation
2. Application
3. Domain
4. Infrastructure

## 各レイヤーの責務

### Presentation

- リクエスト受付
- 入力検証
- 認証、認可
- レスポンス整形

ここには業務判断を置かない。

### Application

- ユースケースの実行順序を定義する
- 複数のドメインオブジェクトや外部サービスを調停する
- トランザクションや副作用の流れを制御する

### Domain

- 業務ルールを保持する
- エンティティ、値オブジェクト、集約を定義する
- 不変条件を守る

### Infrastructure

- データベース
- ストレージ
- 外部 API
- 認証基盤
- メッセージング

技術的な詳細はこの層に閉じ込める。

## 依存関係のルール

- 外側の層は内側に依存してよい
- 内側の層は外側の技術詳細を知らない
- 外部サービスとの接点はインターフェースで抽象化する

## 実務上の判断基準

- ドメインルールが HTTP や ORM の型に触れていたら境界が壊れている
- ユースケースが SQL や SDK 呼び出しを直接持っていたら分離が弱い
- UI の都合でドメインモデルが変形し始めたら DTO や ViewModel を導入する
