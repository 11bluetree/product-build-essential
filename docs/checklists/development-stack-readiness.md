# Development Stack Readiness Checklist

POC が終わり、本開発するにあたっては以下のスタック選定が決まっている状態とする。

## 判断の前提

- 開発開始からデプロイまでの主要スタックが ADR で 1 つずつ記録されている
- 各 ADR に採用理由、却下案、再評価条件が残っている
- 未決事項を残したまま実装を始めない

## 開発基盤

- [ ] パッケージマネージャが決まっている
- [ ] ランタイムとバージョン管理方法が決まっている
- [ ] モノレポ管理やワークスペース運用方法が決まっている
- [ ] ビルド方法が決まっている
- [ ] 環境変数の管理方針が決まっている

## アプリケーション構成

- [ ] フロントエンドの主要フレームワークが決まっている
- [ ] バックエンドの主要フレームワークが決まっている
- [ ] API 契約の表現方法が決まっている
- [ ] CSS ライブラリまたはスタイリング方針が決まっている
- [ ] UI コンポーネント方針が決まっている
- [ ] データ取得と状態管理の方針が決まっている

## 品質基盤

- [ ] 単体テストのフレームワークが決まっている
- [ ] 結合テストの進め方が決まっている
- [ ] E2E テストの要否と採用する場合のフレームワークが決まっている
- [ ] Linter と Formatter が決まっている
- [ ] 型チェックの方法が決まっている
- [ ] pre-commit と pre-push による品質汚染防止の方針が決まっている
- [ ] 秘匿性の高い情報の管理方法が決まっている
- [ ] 秘匿性の高い情報をコードベースに残さないための方針が決まっている

## データと運用

- [ ] データストアが決まっている
- [ ] ORM またはマイグレーション方針が決まっている
- [ ] 認証、認可の方式が決まっている
- [ ] デプロイ先とデプロイ方法が決まっている
- [ ] CI/CD の構成が決まっている
- [ ] 監視、ログ、エラー通知の方針が決まっている

## 参照先

- 技術判断の記録方法は `docs/decisions/adr-template.md` を参照する
- 判断対象の考え方は `docs/decisions/README.md` を参照する
