# Design Documents

実装前の詳細設計を記録するディレクトリ。

## 役割

- **何のためのディレクトリか**
  - ADR（`docs/decisions/`）で「何を使うか」を決めた後
  - 実装開始前に「どのように実装するか」を詳細設計する

- **対象の粒度**
  - 1 機能 = 1 設計ドキュメント
  - 複数のエンドポイントやコンポーネントを含む場合もある

## 記録のルール

1. 1 機能 1 ディレクトリで管理する
   - ディレクトリ名は機能・フィーチャー名（例：`user-authentication/`、`order-checkout/`）

2. 各ディレクトリ配下に複数の詳細設計ドキュメントを配置する
   - `design-doc.md` - 機能全体の設計概要
   - `api-spec.md` - API エンドポイント仕様
   - `db-schema.md` - データベーススキーマ設計
   - `ui-flow.md` - UI フロー、状態遷移
   - その他、必要に応じて

3. 設計完了後のライフサイクル
   - PR でレビュー（実装前に承認）
   - マージされたら実装開始
   - 実装で発見があれば設計ドキュメントも合わせて更新

## 設計ドキュメントに含めるべき内容

### 機能設計（design-doc.md）

```
# [機能名] 設計書

## 概要
- 機能の目的と背景
- 対象ユーザーと利用シーン

## 前提と制約
- ADR への参照（どの技術決定を活用するか）
- 非機能要件（パフォーマンス、セキュリティ要件）
- 実装上の制約

## 主要な設計判断
- 責務分離（レイヤー間の責務）
- ドメインモデル（エンティティ、値オブジェクト）
- エラーハンドリング戦略

## 実装での検討事項
- 実装時に判断が必要な項目
- 未確定の部分
```

### API 仕様（api-spec.md）

```
# API 仕様

## エンドポイント一覧
- POST /api/users/authenticate
- GET /api/users/{id}
- ...

## 各エンドポイント詳細
- リクエスト形式
- レスポンス形式
- エラーレスポンス
- 認可・認証要件
```

### DB スキーマ（db-schema.md）

```
# データベーススキーマ

## テーブル一覧
- users
- sessions
- ...

## テーブル詳細
- カラム定義
- インデックス戦略
- 物理制約（NOT NULL、UNIQUE など）
- 関連性
```

## テンプレート

詳細なテンプレートは `docs/playbooks/design-doc-template.md` を参照する。

## 設計レビューの観点

- [ ] 責務分離が適切か（`docs/core/` の原則に準拠）
- [ ] API 契約が決定されているか
- [ ] DB スキーマにドメイン知識が反映されているか
- [ ] エラーハンドリングが曖昧でないか
- [ ] セキュリティ要件が実装設計に反映されているか（`docs/quality/security-baseline.md` 参照）
- [ ] テスト戦略が設計に含まれているか（`docs/quality/testing-strategy.md` 参照）
